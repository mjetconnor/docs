Entity Results Elements

Because entities are persistent and clearly identified by their entity IDs (aka their "eid"), we have some options about which properties to include in a response.  If you ever need more data related to an entity, you can always do another GET query using the entity's eid.

With this in mind, there are three rules that apply to determine which property values are included in the response to a query -- full, summary, and fields.

One of two rules apply by default depending only on the level of the entity within the hierarchy of the response.  For the top entity in the result, the full inclusion rule applies.  For all other entities, in other words all nested entities, the summary inclusion rule applies.  Nested entities are those entities that are included in a response because they are a property value or element of something else in the same response.

When the full rule applies (for top level entities), every property that has a value for the entity is included in the response.  All properties that have a value, whether or not they are documented, are included.  Conversely, no property is included if it has no value, in other words if its value is null, whether or not they are documented as potential properties of that entity.  Consequently, if a property does not appear in a full response context, it may safely be concluded that the entity has no value for that property.  And conversely, it is always legal for property values that you may not expect to appear in a full response.  If you don't need them, just ignore them.

When the summary rule applies (for nested entities), the response only includes the entity's eid and type properties, and at most one of these also -- title, name or url. The precedence order is the one given here, title being the most preferred and url being the least.

Regardless of the rule or context, the result data for every entity will include the eid and type properties.  Always.  You can always tell if a value in a response is an entity or non-entity object by whether there is an eid property.  If there is no eid property, it is not an entity.

If a query specifies the fields query parameter, the response is constructed accordingly.  This is the fields inclusion rule.  The "fields" query parameter is used to declare exactly how a response should be denormalized.  At its simplest, the value of this parameter is a comma delimited list of property names.  The top level entity in the response will include those properties exactly.  (The eid and type  properties are also included, as they always are for entities.)
