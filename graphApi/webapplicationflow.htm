Web Application Flow

From the OAuth 2.0 v20 specification:

A web application is a confidential client running on a web server.  Resource owners access the client via an HTML user interface rendered in a user-agent on the resource owner's device. The client credentials as well as any access token issued to the client are stored on the web server and are not exposed to or accessible by the resource owner.

In other words, use this flow if your application is a web application that executes in a server under your control, but where the ultimate consumer is a user acting through a user agent like a web browser.  What signifies this kind of flow is that the access token is ultimately going to your application server, but it flows through the user's browser.  Despite this long-winded explanation, this is probably the most common and familiar use case.

 

This flow strings the OAuth 2.0 requests together in the following way.

Step 1.  From your end-user's user agent, execute an /oauth/authorize?response_type=code request to an appropriate MindjetGraph authentication URI.  Specify a redirect URI that is handled by your application server.  Use the state parameter if necessary to coordinate responses to requests.

The execution of this request may itself redirect once or even twice through the user agent, if there is a need to bring up login and authorization screens.  If the user does not have an active Mindjet session, they will be asked to log in.  Then, if the user has not already authorized your application to run on their behalf, they will be asked to do so.  Once there is an active session and authorization in place, the final redirect with the authorization code is sent.  Note that if there is already an active session, and the user has already authorized your app -- the most common case -- none of these interim redirects are required, and the redirect to your app is sent immediately.

Lastly, the hostname in that redirect URI is checked against the value that has been registered with the application.  This is done to protect against spoofing of requests.

Step 2.  An authorization code is returned to your application by way of the redirect URI in Step 1.  Using this code, your application code, running on your server, executes an /oauth/token request, specifying the grant_type=authorization_code parameter in the request body.  Note that you specify the client secret in this request, but that's OK because the request is encrypted by virtue of using https:.

Step 3.  Get the access token from the response to Step 2.  Make REST requests by specifying that access token in the "Authorization" header of each request.
