Collections and Plural Property Names

Some properties are defined with a multiplicity greater than 1.  This means that a given object having that type may have more than one value for that property.  Another way of looking at this is that the object has exactly one value for that property, and that value is a collection of those otherwise distinct properties.  

Take for example the Group class, that has the member property, defined with a requirement level of "[ * ]", or "0..*" in UML terms.  From a graph or RDF perspective, this means that a given resource, one of type Group, might have multiple member properties defined on it.  More precisely, there may be multiple RDF statements in the graph that have that Group resource as the subject, sief:member as the property, and only differ by the agent resources that are the objects for each statement.  The following diagram attempts to illustrate this:

In RDF N-Triple syntax, this might look like this:

_ds:aGroup  sief:member  _ds:agent-1 .

_ds:aGroup  sief:member  _ds:agent-2 .

_ds:aGroup  sief:member  _ds:agent-3 .

_ds:aGroup  sief:member  _ds:agent-4 .

This can be quiet inconvenient for APIs like REST queries, especially those that use JSON format, because JSON (or similar syntaxes) cannot handle having multiple instances of a property for a single object.  It is more convenient to think of the object as having a single property value, but the value of the property is a collection of all of the individual property values, as illustrated here:

This collection is not a physical object in the system, but only an artifact created for generating query responses.

Notice in that last diagram that we used the property name "members", the plural of "member".  This is how we keep these two ways of viewing the graph in order.  When the singular form of a property name is used, the value of that property will be a single object.  When the plural form is used, the value will be a collection made by aggregating together all of the individual properties that have the matching singular name.

The table in Properties Index delineates all of the singular and plural name combinations used in MindjetGraph.

It is typically the case that when operating in an RDF or SPARQL environment, the plural form is never used.

In REST queries, the plural form of property names is very important.  Queries that specify the plural form indicate that a collection is expected in the result.  The greatly simplifies the job of the developer to know what to expect in a response and how to handle it.  See the chapter on plural property names in the REST API reference for more details about that.
