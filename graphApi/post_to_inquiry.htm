Posts to Inquiry Entities

Post requests are handled as defined here when the target is an entity of type Inquiry.  Please keep in mind that Challenge and SuggestionBox are sub-classes of Inquiry.  All of the posts described here apply to all kinds of inquiries, including those two.

The appropriate response is determined by the action and type post properties.

The uiForm property of the inquiry's governor indicates what template properties have been configured for the corresponding SpigitEngage community.  Additional post properties MAY be included where the post property's name corresponds to a field name in the uiForm.  If included, the post property's value MUST be consistent with the field's fieldType.  If the form field is marked as required, then it MUST be included in this post.

This property must be specified in order to distinguish this kind of vote from others.  Note that the definition for the PairwiseBallot class has no MUST properties, so that the value required for this post property is simply:

{ type : "PairwiseBallot" }

up

Idea

MUST

The idea being voted up.

down

Idea

MUST

The idea being voted down.

Returns:  The created Vote object.

For the created vote object, the benchmark property will be a PairwiseBallot.  The valuation property will be a PlainObject with the properties up and down, whose values are the ideas given in the post.  In short, the returned vote object just echoes back the data given in the post.

 

Get a set of pairwise ballots

Added v3.7

Post property

Datatype

Req

Description

action

String

MUST

"getPairwiseBallots"

count

Integer

MAY

The number of ballots to get.  Default is 1.

Returns:  A collection of PairwiseBallots.

The requested number of ballots is returned as specified by the count post property, or up to the maximum number of ballots remaining.  If an empty collection is returned, this indicates that no unvoted pairs remain for the current user.

The returned pairwise ballots will always include values for the "left" and "right" properties.  It is important that an application that renders a pairwise voting screen always locates the left and right elements consistently.  It is not OK for the application to change the order, for instance by sorting the two ideas by ID, title, created date or some other attribute, and then displaying accordingly.  It is entirely possible that the same two ideas may appear in a later ballot but reversed.  If so, they MUST also be rendered in their correspondingly reversed locations.

 

Get a prediction questionnaire

Added v3.7

Post property

Datatype

Req

Description

action

String

MUST

"getPredictionQuestionnaire"

Returns:  A Questionnaire object.

Get the information needed to present a predictions query to a user.  This information takes the form of a Questionnaire object.  Only one prediction questionnaire is returned at a time.  This questionnaire SHOULD be "voted on" before getting another one.  That is done by posting a prediction review to the idea named as the object of this questionnaire.

The questionnaire will have these properties:

Property

Datatype

Description

form

Form

This form provides all of the information necessary to render a UI form to prompt the user for their prediction values.  The fields in the form (FormField type values)  provide the property name, display label and value data types for each of the prediction questions.  Note that for the field data types of Money and Interval, which are common in prediction forms, the field's unit property identifies the currency or time unit, resp, to use when rendering the form.

 

The promptStyle for all the fields in this form will have the value "higherLower", indicating that the response to the form, ie. the valuation, MUST contain one of the two values "higher" or "lower".  These values are made in the context of the corresponding basis values, as given in the basis property in this questionnaire.

 

Predictions forms are entities.  The same form typically appears repeatedly for predictions questionnaires within an inquiry.  It is possible then, and probably a good idea, to cache this value locally if the application is repeatedly prompting the user for predictions.

object

Idea

This is the idea that SpigitEngage wants the user to evaluate next.

basis

PlainObject

These are the basis values to use when prompting for a predictions evaluation.  The property names in the basis object correspond to field names in the form.  The values of those properties are the bases in relation to which the user is asked to vote "higher" or "lower".

 

Although the predictions form is relatively long-lived, the basis values are constantly recalculated by the system in order to hone in on final values.  It is important that fresh questionnaires are used to prompt users for prediction votes so that these basis values are reasonably current.
