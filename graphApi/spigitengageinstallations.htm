Spigit Engage Installations

Each SpigitEngage installation, also known as an instance, is a dedicated set of services and a data repository that supply the infrastructure for SpigitEngage in support of a specific customer.    The instance is addressable via a "spigit.com" sub-domain or in some cases via a vanity URL.  For example:

Spigit Sub-domain

https://innovation.mycompany.spigit.com

Vanity URL

https://innovation.mycompany.com

Vanity URLs are redirected to corresponding "spigit.com" sub-domains.

In the MindjetGraph data model, instances as a whole are represented by Tenant entities.

SpigitEngage instances have multiple entry points, each with distinct membership rules and access controls.  SpigitEngage refers to these access points as communities.  Communities can be designed to address different audiences, agendas, geographical locations, business lines, etc.

Every community has a distinct sub-domain within the instance.  If a user navigates to that URL in their browser, they are taken to the home page for the community.  The primary community is typically indicated by the "www." sub-domain, such as: http://www.mycompany.spigit.com.  Some SpigitEngage instances are set up with a different sub-domain as the primary community and may not use "www." at all.  As additional communities are added, a different sub-domain is allocated for each.  For example:

http://mycommunity.mycompany.spigit.com

Typically, users will be asked to use a specific community as their primary entry point to SpigitEngage.  From that home base, a user is free to visit other communities for which they have been authorized to access or established as a member.

Because communities define different access mechanisms, the community URLs provide the starting point for authentication.  The OAuth 2.0 pages refer to these starting  point URLs as authorizationServiceURLs.  For SpigitEngage, these authorizationServiceURL are the URLs for the communities in an instance.

If a user already has a SpigitEngage session active in their browser, the OAuth 2.0 flow will simply recognize that user as the authenticated user.  But if there is not an active session, the authentication process is the one dictated by the community used as the authorizationServiceURL.  Some communities may use username/password login pages, others may be associated with various SAML or LDAP connectors, all within the same SpigitEngage instance.  It is important that authentication begins with the same community URL that a user uses to access the SpigitEngage web app in their normal usage.
