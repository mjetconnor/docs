Requirement Levels and multiplicity

In RDF, any given resource can have any number of relationships with any given property with other resources.  This is powerful as a data model, but can present a challenge to constructing applications that are simple, maintainable, reliable and predictable. To that end, all properties in MindjetGraph are defined with a "requirement level".  This is a cue to a MindjetGraph app developer as to how to code for accessing those properties -- which properties are always present, which require checks for null, which are common, and which may have multiple values.

The “Req” column in each class definition page declares the requirement level for the properties.  These requirement levels may be thought of as indicating a UML-style  multiplicity characteristic.  They also correspond to the ISO 2119 standard terminology of  “MUST”, “SHOULD” and “MAY” as defined by the W3C in RFC 2119.

It is important to know that it is always legal for undefined or unknown properties to exist for an object in addition to the ones explained here.  The graph constantly grows with the addition of new knowledge as represented by new datatypes and new relationships.

 

* These are the values that appear in the MindjetGraph class definitions.

"Req" values *

RFC 2119

UML Multiplicity

Description

MUST

Yes

1

The property MUST have exactly one value.  There is no need to check for null values for this property.  Objects of the class cannot exist without this property.

SHOULD

Yes

0..1

The property SHOULD have exactly one value.  It MAY have no value and MUST have no more than one value.  

MAY

Yes

0..1

The property MAY have one value and MAY have no value.   It MUST have no more than one value.

From an implementer's point of view, the distinction between SHOULD and MAY is not very substantial.  In both cases, one must check for possible null values.

[ * ]

No

0..*

There MAY be zero or more values to the property.  In other words, the value of the property could be thought of as a collection of any size.  That collection may be empty.

[ 1 ]

No

1..*

There MUST be at least one value to the property and there MAY be more.  In other words, the value of the property could be thought of as a collection of any size but that must have at least one element.

RULE

No

0..*

Values for this property are defined by a rule and derived from other concrete property values.  Common rules are inversion and transitivity.

An inversion type of rule goes like this.  If there is a property that has a value like this: [ entityA propertyX entityB ], then we can infer that there is also this property: [ entityB propertyY entityB ].  An example of this is the memberOf property on Agent, that is the inverse of the member property on Group.  The member property is concrete; the memberOf property is inferred from it.
