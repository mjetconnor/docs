PUT Queries

Use the HTTP PUT request method to modify the properties of an existing MindjetGraph entity.

The URL to be used for HTTP PUT requests is exactly the same described for the GET method.  As with GET queries, the basic URL is simply an entity ID.  That URL may be extended with property segments and matrix parameters to indicate an object that is somehow derived from the entity of the root URL.

GET requests have no request body.  PUT requests do.  The request body is simply a set of name/value pairs in the format specified by the Content-Type header, that provides updated property values for the entity identified by the query URL.  The name of a name/value pair is the name of a property for that entity.  The value is the new value to set for that property, overriding the old value if necessary.

Application logic always applies to update operations like this.  The setting of a property may be rejected for access or logic reasons, or may trigger side-effects beyond that one simple action.  PUT requests are atomically transactional, meaning that all of the property updates embodied in the request will either be successful or rejected together as a unit.

The W3C defines the PUT method as being "idempotent", and so it is with the MindjetGraph REST API.  Idempotency means that the operation is one that has no additional effect if it is called more than once with the same input parameters.  In the case of a PUT in MindjetGraph REST, that just means that the same value will be reassigned to a given property.  If there are side-effects, they will not be replicated beyond the initial setting.

Example:

PUT https://www.example.mindjet.com/api/ds1/node-75494

with headers:

Content-Type: application/json

Authorization: Bearer 914a9b4d83fa7333d3b517643a30bac2

and request body:

{

   "title":"This is my new title",

   "img":"http://mystuff.example.com/images/24393310.jpg"

}

sets new values for the title and img properties of the entity "_ds:node-75494".
