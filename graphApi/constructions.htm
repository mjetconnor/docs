Object Constructions

Formatted Text and Enumerations

The values of some datatypes are text string values that are constrained in some way by specification.

Enumerated values are of this type, where the legal values for the datatype are delineated in some kind of list.  Some enumerated types are defined by industry standard, such as the ISO 639-2 standard that defines Language values.  Other enumerated types are defined by this model, such as for the EntityStatus class.  

Other formatted text types are defined more in the manner of a text pattern.  Examples of this are the EmailAddress and PhoneNumber classes.  Email addresses are tightly spec'd by industry standards RFC 5321 and 5322, whereas phone numbers are still largely by informal common usage.

Objects

All non-primitive values are objects.  Objects are really just collections of name/value pairs, where the name is a property name and the value is the value of that property.  The specific physical representation depends on the language binding.  This corresponds very naturally to the JavaScript notion of objects as associative arrays, for instance.

When objects are represented as associative arrays or name/value pairs, as with JavaScript as we just mentioned, you cannot have the same property name represented twice.  This is  handled by using the singular or plural form of a property name.  This is discussed in more detail in Collections and Plural Property Names.

Many of the classes defined by the Mindjet Graph  data model describe kinds of objects.  The job of the class definition is to describe what properties are expected to appear for objects of that particular datatype.

Collections

The MindjetGraph data model has no collection datatype per se.  Collections appear as properties that may have multiple values as indicated by the [ * ] or [ 1 ] requirement levels.  The specific form that these objects take depends on the API language binding being used.  The format used by the REST API is discussed here.

Inheritance

Some classes are defined as being sub-classes of other classes.  An example is the Challenge class, which is a subclass of the Inquiry class, which itself is a subclass of both the Collaboration and ConceptScheme classes.

Inheritance simply means that all of the property descriptions defined for the parent class also apply the child class.  In some cases, a child class may re-specify a property found in the parent class in a more restrictive way, for instance by defining a tighter requirement level or by specifying a value datatype that is a subclass of the one specified in the parent.  An example of this last case is the IdeaScore class that is a subclass of the Facet class.  The base property is defined to be a Resource by the Facet class, but is more tightly defined to be an Idea by the IdeaScore class.
