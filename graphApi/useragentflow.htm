User Agent Flow

From the OAuth 2.0 v20 specification:

A user-agent-based application is a public client in which the client code is downloaded from a web server and executes within a user-agent on the resource owner's device.  Protocol data and credentials are easily accessible (and often visible) to the resource owner.  Since such applications reside within the user-agent, they can make seamless use of the user-agent capabilities when requesting authorization.

In other words, use this flow if your application is implemented as something that executes within your user's user agent, for instance as JavaScript in their browser.  This flow is distinct from the Web Application Flow in that the access token never leaves the user agent.  It is retrieve by, and only used in, the user agent.

You can of course cheat by using the flow to get the access token in the user agent and then somehow send the token back to an application server.  It is best not to do that because it opens your application up to being sniffed and hacked.  Better to use the  Web Application Flow if that really is the nature of your application.


This flow strings the OAuth 2.0 requests together in the following way.

Step 1.  From your application, running in your user's user agent, execute an /oauth/authorize?response_type=token request to an appropriate MindjetGraph authentication URI.  Specify a redirect URI that is handled by your application.  Use the state parameter if necessary to coordinate responses to requests.

As mentioned in the Step 1 for Web Application Flow, there may be interim redirects to take care of user login and authentication.

Step 2.  The final redirect is made, hash tagged with an access token response.  The hash tag insures that that part of the redirect URI does not go beyond the user agent, i.e.. the user's browser.

In the case of the User Agent Flow, the page targeted by that redirection must be the page that implements the application.  The "onLoad" script for the page should get the hash tag, for instance from "window.location.hash" in browser JavaScript, and parse out the response elements.

Note that if the page has some server-side components, for instance is a .jsp that is constructed by your server, the hash tag, and therefore the access token in it, will not be shared with the server.  Only script code running in the user agent can see those values.  This is how OAuth 2.0 guards against prying eyes for this kind of flow.

Step 3.  Get the access token from the response to Step 2.  Make REST requests by specifying that access token the "Authorization" header of each request.
